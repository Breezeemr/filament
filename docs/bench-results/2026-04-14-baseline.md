# Filament baseline — 2026-04-14

Full criterium `bench` (not `quick-bench`) on both JDK 21 and JDK 25.
Single machine, back-to-back runs, chain inline fast path enabled,
JDK 21/25 fork-impl split in place.

Hardware: AMD Ryzen AI MAX+ 395 w/ Radeon 8060S, Linux 6.17.0-20-generic
Commit: 98bae4e0e0c0c4c9ac08f9a20da29b47f6ee8b96
Mode: `clj -M:bench -m filament.chain-bench` (full bench)

## Results

| Scenario | JDK 21 | JDK 25 |
|---|---|---|
| Filament chain-10 | 116 ns ±1 | 117 ns ±1 |
| Manifold chain'-10 | 208 ns ±3 | 161 ns ±1 |
| **Ratio chain (Fil/Man)** | **0.55x** ✅ | **0.73x** ✅ |
| Filament chain-10 (no trace) | 216 ns ±3 | 219 ns ±1 |
| Filament zip-4 sleepers | 1.17 ms ±15.5 us | 1.17 ms ±18.4 us |
| Manifold zip-4 sleepers | 1.09 ms ±7.1 us | 1.09 ms ±16.5 us |
| Ratio zip (Fil/Man) | 1.07x ✅ | 1.07x ✅ |
| Filament chain-10 × 1000 concurrent (wall) | 4.39 ms | 4.79 ms |

Both JDKs PASS the <2x bar on both scenarios.

## Observations

- **Filament chain-10 is identical across JDKs (~117 ns).** The inline
  fast path runs on the caller's stack with no `StructuredTaskScope`,
  no vthread submit, and no fork-impl dispatch, so there's nothing
  JDK-specific to measure. The 1 ns spread between 116 and 117 is
  noise.
- **Manifold chain'-10 is 23% faster on JDK 25** (161 vs 208 ns). The
  entire cross-JDK ratio gap is Manifold getting faster, not Filament
  getting slower. Likely general JIT / value-type improvements in
  JDK 25 that the Manifold callback path benefits from and the inline
  Filament path doesn't (because it was already ~one function-call
  deep).
- **Trace capture still costs ~100 ns per chain.** Scenario 3 (no
  trace) clocks ~216–219 ns vs scenario 1's 116–117 ns. Unlike the
  quick-bench oddity, the full bench shows the expected direction —
  no-trace is *faster*, and the cost is a `Throwable` + `fillInStackTrace`.

  But scenario 1 is **faster** than scenario 3. That's because the
  two scenarios take different code paths: scenario 1 hits
  `success-deferred` → inline fast path, scenario 3 wraps the whole
  thing in `(binding [*capture-traces* false] ...)` which forces it
  through `impl/filament` (the bench harness does this deliberately
  to measure the slow-path trace cost). So scenario 3 is actually
  "slow path with no trace" vs scenario 1's "fast path with trace."
  The direct comparison is scenario 3 vs a hypothetical "slow path
  with trace" we don't currently measure.
- **Zip-4 is ~1.07x Manifold on both JDKs.** Identical within noise.
  The 1 ms `Thread/sleep` dominates; vthread submit cost is
  essentially free at this latency.
- **1000 concurrent chains run in ~4.4–4.8 ms wall clock.** ~4-5 µs
  per chain end-to-end including scheduling. No scheduler contention
  up to 1000 vthreads on this machine.

## Future use

These are the numbers to beat. Any change that regresses scenario 1
past ~150 ns or scenario 4 past 1.25 ms is a regression worth
investigating.
